# v0.2 release checklist

This checklist defines the execution-context fidelity release gate. It extends, and does not replace, every automated v0.1 gate in [release-checklist.md](release-checklist.md).

Run the complete local gate from the repository root:

```bash
make verify
```

## Automated behavior and contracts

- [x] `--pid`, `--docker`, and `--podman` are mutually exclusive and reject empty or invalid values.
- [x] A selected PID supplies its network namespace, mount namespace, filesystem root, resolver files, and supported proxy environment.
- [x] Shared namespaces and roots require no elevated capabilities.
- [x] Differing mount/network namespaces and roots report the exact required Linux capabilities.
- [x] Selected `/proc` directories and context descriptors are pinned before any one-way context transition.
- [x] An unreadable selected environment is recorded without preventing network probes.
- [x] A real isolated mount/network/root fixture is entered when the host permits user namespaces.
- [x] An inaccessible cross-user-namespace context fails with a structured `CONTEXT_UNAVAILABLE` error.
- [x] Local Docker and Podman adapters use their documented inspection templates without invoking a shell.
- [x] Remote Docker and Podman services are rejected before a runtime PID can be interpreted through local `/proc`.
- [x] Missing runtimes, failed inspection, malformed or zero PIDs, oversized output, and inspection timeouts fail safely.
- [x] Runtime and iproute2 helpers run in isolated process groups, cannot extend a deadline through inherited output pipes, and retain at most 64 KiB per output stream.
- [x] Route and TCP fan-out preserve one explicit result per resolved address even if an individual task terminates unexpectedly.
- [x] The runtime PID is verified again after context handles are pinned so a concurrent container restart is rejected.
- [x] Container identifiers and runtime errors cannot inject terminal control characters.
- [x] JSON reports identify `process`, `docker`, and `podman` sources and validate conditional PID/container fields against `report.schema.json`.
- [x] Human output identifies the selected execution context and its capability status.
- [x] The full local format, test, coverage, MSRV, package, install, and uninstall gate passes.

## Real-runtime qualification

These checks require disposable Linux hosts or CI runners with the named runtime. Fake-runtime tests remain mandatory but do not replace this qualification.

- [ ] Current Ubuntu LTS with rootful Docker: shared and isolated container contexts pass end to end.
- [ ] Current Fedora stable with rootless Podman: shared and isolated container contexts pass end to end.
- [ ] A stopped container produces `CONTEXT_UNAVAILABLE` in human and JSON modes on both runtimes.
- [ ] Docker over SSH/TCP and remote Podman are rejected on real configured remote services.
- [ ] The documented capability guidance is confirmed for unprivileged, capability-granted, rootful, and rootless cases.
- [ ] Release artifacts pass `make verify` on every supported release architecture.

v0.2 is ready to publish only after every item above is checked with the tested kernel, distribution, architecture, runtime version, and privilege mode recorded in release notes.

## Development qualification evidence

- 2026-07-19: Clean, read-only source mounts in `ubuntu:26.04` (Ubuntu 26.04 LTS, Rust 1.93.1) and `fedora:44` (Fedora Linux 44, Rust 1.96.1) passed `cargo test --all-features --locked` (158 tests), a locked release build, the release binary's structured `TCP_CONNECTION_REFUSED` JSON smoke test, and `cargo package --allow-dirty --locked`. These Docker userspaces validate distribution toolchain and userspace compatibility, but share the CachyOS host kernel and Docker daemon, so they do not satisfy either native-host real-runtime item above.
- 2026-07-19: Docker 29.6.2 on x86_64 CachyOS, Linux 7.1.3. The release binary rejected an unprivileged isolated-container entry with structured capability guidance, then entered the container network namespace, mount namespace, and filesystem root under the required privileges and produced a schema-compatible diagnostic report. The hardened runtime process-group path was rechecked against a disposable Alpine container; the expected `TCP_CONNECTION_REFUSED` report identified the Docker context, and the fixture was removed. This is useful development evidence but does not satisfy the Ubuntu LTS release item above.
- 2026-07-19: Rootless Podman 6.0.1 with runc, netavark, overlay storage, and a 65,536-ID subordinate UID/GID range on the same host. Direct execution rejected isolated namespace entry with structured `CAP_SYS_ADMIN` guidance. `podman unshare` execution entered the container network namespace, mount namespace, and filesystem root and produced a schema-compatible diagnostic report. The hardened runtime process-group path was rechecked with a one-second operation timeout against a disposable Alpine container; the expected `TCP_CONNECTION_REFUSED` report identified the Podman context. The disposable container and image were removed. This does not satisfy the Fedora stable release item above.
